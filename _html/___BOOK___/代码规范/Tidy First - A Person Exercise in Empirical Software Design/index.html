Tidy First - A Person Exercise in Empirical Software Design

译名：整理优先 小改进，大回报，整洁代码设计指南

第一部分：整理

重构需满足不改变行为：
重构的核心原则就是在不改变外部可见行为的前提下，优化代码的内部结构
也就是说重构应该只是优化复杂度/修改命名/拆分整合代码之类"让代码更好看"的操作
如果需要大改，这应该被称为重写


一般来说，使用卫述句Guard Clause是比较好的
但不能是"混合形式"

if(conditon)
{
    // ...
}
// 就可以整理为：
if(!condition) return;
// ...

// 但这种就不合适(可以考虑把condition部分提到函数中)
if(condition)
{
    // some code
}
// other code

无用代码直接删了即可
无需考虑后续是否需要，如果后续需要直接通过版本管理获取即可


代码需要保持一致性，有可能会发生以下情况：
一开始的代码用的是一种简单的方法，后续学会了语法糖就在新的地方用了语法糖
为了保持一致，选用任意一种变体即可，只要是相同的即可


如果接口导致复杂混乱，可以定义理想接口，创建适配器(使用旧接口，实现新接口)，在后续逐步改为新接口(自然废弃/手动替换)后，再内联实现(删除适配器，直接实现)
